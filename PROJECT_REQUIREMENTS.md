# ScreenHarbor 项目需求文档

## 1. 项目概述

**中文名**：幕屿  
**英文名**：ScreenHarbor  
**GitHub 仓库名**：screenharbor  
**产品形态**：Telegram Mini App  

ScreenHarbor 是一个运行在 Telegram 内的影视资料、观影追踪和用户资源投稿平台。用户可以浏览电影、剧集、动画、纪录片等影视条目，记录自己的观看状态，发布评论，并在条目下提交资源说明或链接。

前期目标是以低成本方式快速上线，使用 Cloudflare 免费额度完成 MVP 验证。后期可以将后端业务服务从 TypeScript 重写为 Go。

## 2. 产品目标

- 提供清晰的影视条目浏览体验。
- 支持用户记录想看、在看、看过、弃剧等状态。
- 支持用户在影视条目下提交资源信息。
- 支持资源公开、登录可见、积分可见、仅作者可见等展示规则。
- 建立举报、审核、失效标记机制，降低违规和无效内容风险。
- 通过 Telegraph 承载影视长文介绍、专题文章和公告内容。
- 使用 Cloudflare Workers、D1、R2 完成无自建服务器的前期部署。

## 3. 目标用户

### 普通用户

- 浏览影视简介、评分、标签和资源信息。
- 标记个人观看状态。
- 收藏条目。
- 发表评论和反馈。

### 资源贡献者

- 在影视条目下提交种子、ed2k、网盘、字幕或其他资源信息。
- 填写资源版本、清晰度、字幕、文件大小等说明。
- 设置资源是否公开或是否需要积分解锁。

### 管理员和审核员

- 审核资源投稿。
- 处理举报。
- 下架违规或失效内容。
- 管理影视条目、标签、用户和积分。

## 4. MVP 范围

第一版优先完成核心闭环：

- Telegram Mini App 入口。
- Telegram 用户身份识别。
- 影视条目列表和详情页。
- 影视条目搜索和基础筛选。
- 用户追踪状态：想看、在看、看过、弃剧、重刷。
- 用户评分和短评。
- 资源投稿。
- 资源可见性控制。
- 资源审核。
- 举报和失效标记。
- Telegraph 长文页跳转。

第一版暂不做：

- 复杂会员体系。
- 复杂积分商城。
- Telegram Stars 支付。
- 自动全网资源采集。
- 复杂推荐算法。
- 私信和重社交功能。

## 5. 核心功能

### 5.1 影视条目

每个影视条目需要包含：

- 中文名。
- 原名。
- 英文名。
- 类型：电影、剧集、动画、纪录片、综艺。
- 年份。
- 国家和地区。
- 语言。
- 海报。
- 简介。
- 导演、演员、编剧等演职员。
- 标签。
- 平均评分。
- 评论数量。
- 关联 Telegraph 长文页。

### 5.2 观影追踪

用户可以对每个影视条目标记：

- 想看。
- 在看。
- 看过。
- 弃剧。
- 重刷中。

可选信息：

- 个人评分。
- 当前进度。
- 个人备注。
- 开始观看时间。
- 完成观看时间。

### 5.3 资源投稿

用户可以在影视条目下提交资源信息：

- 资源标题。
- 资源类型：torrent、ed2k、网盘、字幕、其他。
- 链接地址。
- 版本说明。
- 文件大小。
- 清晰度。
- 字幕说明。
- 是否公开。
- 是否需要积分解锁。
- 所需积分。

资源状态：

- 待审核。
- 已发布。
- 已拒绝。
- 已隐藏。
- 已失效。
- 已删除。

### 5.4 评论和评分

用户可以：

- 发布短评。
- 发布评分。
- 回复评论。
- 标记剧透。
- 点赞评论。
- 举报评论。

### 5.5 积分系统

第一版只做基础能力：

- 用户积分余额。
- 积分流水。
- 资源积分解锁记录。
- 管理员手动调整积分。

后续可扩展：

- 签到奖励。
- 投稿奖励。
- 被购买分成。
- Telegram Stars 兑换积分。

### 5.6 审核和举报

必须支持：

- 资源发布前审核。
- 评论举报。
- 资源举报。
- 条目纠错。
- 资源失效标记。
- 管理员处理记录。
- 用户信用分调整。

## 7. 技术方案

### 7.1 前端

- Svelte。
- TypeScript。
- Vite。
- Telegram WebApp SDK。

前端负责：

- Mini App 页面展示。
- 用户交互。
- 表单提交。
- 状态展示。
- 调用 Cloudflare Workers API。

### 7.2 后端

- Cloudflare Workers。
- TypeScript。

后端负责：

- Telegram initData 校验。
- 用户创建和登录态识别。
- 影视条目 API。
- 追踪状态 API。
- 评论 API。
- 资源投稿 API。
- 审核 API。
- 积分 API。
- Telegraph API 对接。

### 7.3 数据库

- Cloudflare D1。

D1 存储：

- 用户。
- 影视条目。
- 标签。
- 演职员。
- 用户追踪记录。
- 评论。
- 资源投稿。
- 积分流水。
- 举报。
- 审核日志。
- Telegraph 页面映射。

### 7.4 文件存储

- Cloudflare R2。

R2 存储：

- 海报。
- 封面。
- 用户上传图片。
- 资源说明截图。
- 轻量附件。

### 7.5 长文内容

- Telegraph。

Telegraph 负责：

- 影视长介绍。
- 专题文章。
- 版本说明。
- 公告。
- 使用说明。

Telegraph 不负责：

- 用户系统。
- 权限控制。
- 积分。
- 评论。
- 资源审核。
- 业务主数据。

## 9. 权限规则

用户角色：

- user：普通用户。
- contributor：贡献者。
- reviewer：审核员。
- admin：管理员。

资源可见性：

- public：公开。
- login_only：登录后可见。
- author_only：仅作者可见。
- review_only：审核中可见。
- points_only：积分解锁后可见。

资源价格：

- free：免费。
- points：积分解锁。
- membership：会员解锁，后续扩展。

## 10. 合规和风控

资源投稿必须有审核和举报机制。平台应明确要求用户只发布合法授权内容、自有内容或允许传播的资源信息。

第一版需要支持：

- 敏感内容举报。
- 资源失效举报。
- 管理员下架。
- 用户封禁。
- 审核日志。
- 删除日志。
- 资源修改记录。
